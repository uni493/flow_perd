# 广告流量预测论文洞察报告

资料核验时间：2026-07-06  
报告范围：平台/投放侧广告流量预测，服务模型选型与落地  
适用对象：广告平台、DSP/SSP、媒体变现、投放优化、MPC/预算控制团队

## 0. 执行摘要

广告流量预测不是一个单纯的「未来时序点预测」问题。论文脉络显示，它至少由五类问题拼在一起：

| 主线 | 预测对象 | 典型决策 |
| --- | --- | --- |
| 受众/库存预测 | targeting 组合下的可达用户、可用展示、设备 reach | 是否接单、保量库存评估、售前报价 |
| 保量投放与 allocation | 未来 user visit/impression 的供给和匹配关系 | 合约是否能按期交付、每次请求分给哪个订单 |
| RTB bid landscape | 每次竞价的 winning price/market price 分布 | 出价、预算利用率、 win-rate 与成本控制 |
| pacing 与预算控制 | 未来消耗机会、花费节奏、约束风险 | 是否压速、提速、保 ROI/ROS、避免预算提前耗尽 |
| 收入/成本/效果预测 | revenue、CPC、CTR/CVR、ROAS | 经营预测、预算分配、策略解释 |

最重要的落地判断如下：

1. **流量预测要先拆清“自然机会量”和“被系统控制后的观测量”。** 如果 `req_cnt`、展示量或消耗已经被预算、pacer、审核、素材、出价策略污染，直接拿当日累计比例校正未来流量，会把控制系统造成的低量误判为市场供给下降。这个结论来自 allocation/pacing 论文的共同约束，也与 RTB 竞价分布论文强调的 censoring/partial observation 一致。
2. **长尾、高维 targeting 组合不能只靠深度时序模型。** 对受众/库存规模，采样、频繁项集挖掘、HyperLogLog、MinHash 等 sketch 方法更像基础设施；Transformer 更适合作为充足样本上的多变量残差模型，而不是替代高维集合估计。
3. **RTB 场景应预测分布而不是点估计。** winning price 只有赢标时完整可见，输标时只知道下界，因此需要 censored regression、survival analysis 或分布建模；点预测 CPC/成本不够支撑出价决策。
4. **allocation 和 pacing 论文都把 forecast 当成“先验”，而不是一次性真值。** 好系统要能在预测准确时吃到收益，在预测错、突发流量 spike、预算小样本波动时仍然稳。
5. **如果当前目标是 WAPE/MAE，P50 是自然输出；如果下游 MPC 的高估/低估代价不对称，应提供 P50/P70/P80 等分位数。** 这是统计决策论和 pacing/预算控制场景共同导向的工程结论。

## 1. 问题边界：这里的“广告流量预测”到底指什么

本报告聚焦平台/投放侧流量预测，不把 CTR/CVR 当主线。本文里的流量包括：

- 广告请求量、可竞价 bid request、可展示 impression opportunity。
- 定向条件下的受众规模、设备 reach、库存可达量。
- 保量合约下未来可分配 visit/impression。
- RTB 市场中的 winning price 分布、win-rate 曲线、竞价环境。
- pacing/MPC 下「剩余机会量」「剩余可消耗空间」「未来成本环境」。

CTR/CVR/收入/CPC 不是主预测对象，但它们会影响流量利用价值，因此作为相邻模块纳入。

一个实用的系统边界是：

```text
自然供给预测
  -> 定向/库存/受众规模估计
  -> 竞价环境与成本分布预测
  -> allocation / pacing / MPC 决策
  -> 受控后的展示、消耗、转化观测
  -> 当日反馈校正
```

关键是不要把最右侧「受控后的观测」简单当成最左侧「自然供给」。

## 2. 论文脉络与核心洞察

### 2.1 受众/库存预测：高维组合、长尾和集合估计

**代表论文**

| 论文 | 链接 | 贡献 |
| --- | --- | --- |
| Scalable Audience Reach Estimation in Real-time Online Advertising | [arXiv:1305.3014](https://arxiv.org/abs/1305.3014) | 用分布式、容错、多变量分层采样来快速估计 targeting 条件下 reach，强调 fuzzy fallback 处理少数群体 |
| Forecasting Granular Audience Size for Online Advertising | [arXiv:1901.02412](https://arxiv.org/abs/1901.02412) | 改造 Eclat/FIM 适配 categorical targeting 组合，再结合时序和条件概率预测细粒度 audience size |
| Real-Time Device Reach Forecasting Using HLL and MinHash Data Sketches | [arXiv:2502.14785](https://arxiv.org/abs/2502.14785) | 用 HLL/MinHash sketch 支持多维聚合和交集的实时 reach 估计，目标是替代慢 SQL join |

**洞察**

受众/库存预测的难点不是普通日级趋势，而是「维度组合爆炸 + 长尾稀疏 + 需要实时查询」。例如 `{US, Chrome, Mobile}` 只是一个很小的组合，真实广告系统还会叠加地域、媒体、行业、人群包、频控、设备、时段等条件。  

这类问题的合理架构通常不是「把所有组合喂给一个大模型」，而是：

- 对常见组合建立可查询的聚合、样本或频繁项集。
- 对稀疏组合使用 fallback、条件概率分解或相似人群近似。
- 对大规模集合交并补使用 sketch，接受可控误差换取实时性。
- 时序模型只负责在已有可估计组合上预测趋势、季节性和异常修正。

**对落地的启发**

对于广告流量预测系统，如果任务粒度包含大量定向组合，建议先建设“库存/受众估计层”：支持按组合查询历史 reach、请求、展示和不确定性。没有这个层，直接训练深度模型容易在长尾上看起来拟合、实际不可用。

### 2.2 保量投放与 allocation：预测服务于分配，而不是孤立指标

**代表论文**

| 论文 | 链接 | 贡献 |
| --- | --- | --- |
| Ad Serving Using a Compact Allocation Plan | [arXiv:1203.3593](https://arxiv.org/abs/1203.3593) | 面向 guaranteed contracts，用离线 compact allocation plan 支撑线上低延迟、分布式、无状态 ad serving |
| Online Allocation with Traffic Spikes: Mixing Adversarial and Stochastic Models | [arXiv:1711.05764](https://arxiv.org/abs/1711.05764) | 将可预测随机流量和不可预测 adversarial spike 混合建模，避免纯随机假设过于乐观 |
| Online Ad Allocation with Predictions | [arXiv:2302.01827](https://arxiv.org/abs/2302.01827) | 将机器学习预测引入 online allocation，并要求预测好时提升、预测差时保持鲁棒 |

**洞察**

保量广告里的预测不是报表，而是约束优化的输入。`Compact Allocation Plan` 的核心背景是：合约提前售卖时只能依赖未来 user visit 的 forecast，但线上请求必须毫秒级决策。`Traffic Spikes` 进一步说明，真实广告流量不满足干净的 i.i.d. 或随机顺序假设，突发新闻、热点事件、平台策略变化都会制造 spike。  

因此，一个好的预测系统要输出的不只是 `point forecast`，还应输出：

- 每个合约/任务/定向组合的供给置信度或分位数。
- spike、节假日、活动、异常流量的风险标识。
- 不同预测误差对分配失败、欠投、过投的决策影响。

**对落地的启发**

如果流量预测用于 MPC 或 pacing，不应只追求全局 WAPE。要同时回放「给定预测后，分配/出价/压速策略是否更好」。预测误差在高预算、高价值、临近 deadline 的任务上应有更高权重。

### 2.3 RTB bid landscape：竞价环境要建模分布和 censoring

**代表论文**

| 论文 | 链接 | 贡献 |
| --- | --- | --- |
| Display Advertising with Real-Time Bidding (RTB) and Behavioural Targeting | [arXiv:1610.03013](https://arxiv.org/abs/1610.03013) | RTB 综述，覆盖用户响应、bid landscape、出价、收入优化、动态定价等 |
| Deep Landscape Forecasting for Real-time Bidding Advertising | [arXiv:1905.03028](https://arxiv.org/abs/1905.03028) | 用 RNN 建模不同 bid price 下的条件赢标概率，并用 survival/censoring 思路处理只部分可观测的 market price |
| Scalable Bid Landscape Forecasting in Real-time Bidding | [arXiv:2001.06587](https://arxiv.org/abs/2001.06587) | 放松固定方差、单峰分布等假设，提出 heteroscedastic censored regression 和 mixture density censored network |

**洞察**

RTB 里的“流量”有两层含义：有多少竞价机会，以及以什么价格能赢下这些机会。第二层比纯请求量更直接影响预算消耗和 ROI。  

关键挑战是 censoring：DSP 只有赢标时能看到 winning price；输标时只知道「市场价高于自己的 bid」。如果忽略这个观测偏差，模型会系统性低估市场价格，导致出价策略偏激或预算利用异常。

**对落地的启发**

当预测结果要进入 MPC/出价系统时，建议至少补充三类输出：

- `P(win | bid, context)` 或 win-rate 曲线。
- winning price 分布的分位数，而不是单点 CPC。
- 数据 censoring 标识：赢标样本、输标下界样本、缺失价格样本分开训练或加权。

### 2.4 pacing 与预算控制：预测只是先验，反馈控制决定稳定性

**代表论文**

| 论文 | 链接 | 贡献 |
| --- | --- | --- |
| A Field Guide for Pacing Budget and ROS Constraints | [arXiv:2302.08530](https://arxiv.org/abs/2302.08530) | 比较预算 pacing 与 ROS pacing 的解耦、min-pacing、fully-coupled dual-based 算法 |
| Dynamic Budget Throttling in Repeated Second-Price Auctions | [arXiv:2207.04690](https://arxiv.org/abs/2207.04690) | 分析 repeated second-price auctions 中 throttling 的 regret 和 competitive ratio，并比较 pacing/throttling |
| Feedback Control for Small Budget Pacing | [arXiv:2509.25429](https://arxiv.org/abs/2509.25429) | 将 bucketized hysteresis 与 proportional feedback 结合，面向小预算 campaign 改善 pacing 稳定性 |

**洞察**

pacing 论文共同说明：预算控制不是“预测未来流量后按比例花钱”这么简单。真实系统还要同时处理预算约束、ROI/ROS 约束、竞价波动、小预算离散性、反馈滞后和参数稳定性。  

`Field Guide` 的价值在于把预算 pacing 和 ROS pacing 的组合方式形式化；`Feedback Control` 则提醒小预算场景下，少量 auction 的随机性会让简单比例控制变得不稳定。

**对落地的启发**

对 MPC/剩余流量预测，建议预测层提供：

- 剩余机会量 P50/P70/P90。
- 剩余可消耗金额或可赢量的分布。
- 当前 pacing 状态、预算耗尽风险、控制污染标识。
- 预测误差和控制误差分离评估：模型预测错，还是控制策略导致观测偏离。

### 2.5 收入、成本与 CTR：相邻模块可以借方法，但不要混同目标

**代表论文**

| 论文 | 链接 | 贡献 |
| --- | --- | --- |
| Online Advertising Revenue Forecasting: An Interpretable Deep Learning Approach | [arXiv:2111.08840](https://arxiv.org/abs/2111.08840) | 用 TFT 预测 publisher 广告收入，引入其他 publisher 收入作为协变量并做可解释分析 |
| Interpretable Deep Learning for Forecasting Online Advertising Costs | [arXiv:2302.05762](https://arxiv.org/abs/2302.05762) | 预测日均 CPC，强调竞争对手 CPC pattern 的聚类协变量和 TFT 可解释性 |
| Feature Representation Learning for Click-through Rate Prediction: A Review and New Perspectives | [arXiv:2302.02241](https://arxiv.org/abs/2302.02241) | CTR 特征表示学习综述，梳理哪些特征、如何 embedding/representation |

**洞察**

收入、CPC、CTR 论文对流量预测最大的贡献不是目标本身，而是：

- 多变量协变量能显著提升广告时间序列预测，尤其是行业、竞品、大盘、相似 publisher/task。
- TFT 等模型在「静态特征 + 历史序列 + 已知未来特征」混合输入时很自然。
- 可解释性在广告系统里很重要，因为业务需要知道预测变动来自节假日、大盘、竞价环境还是自身策略。

**对落地的启发**

如果当前流量预测只有任务历史曲线，下一步最值得加的不是更大的模型，而是大盘/分层/竞价/控制状态协变量。

| A Time Series is Worth 64 Words: Long-term Forecasting with Transformers | [arXiv:2211.14730](https://arxiv.org/abs/2211.14730) | PatchTST 用 patch 和 channel-independence 提高长历史建模效率 |

**洞察**

通用时序模型值得作为 benchmark 和特征学习工具，但广告流量有几类它们默认不解决的问题：

- targeting 组合的集合估计和长尾补全。
- 出价、预算、pacer 造成的观测污染。
- RTB winning price 的 censoring。
- allocation/pacing 下的非对称损失。
- 业务策略变更导致的概念漂移。

**对落地的启发**

深度时序模型的合理位置是「在强 baseline 之上学习残差/后验校正」，而不是一上来替代库存估计、采样、sketch 和控制逻辑。

## 3. 核心论文卡片

### 3.1 Scalable Audience Reach Estimation in Real-time Online Advertising

- 来源：[arXiv:1305.3014](https://arxiv.org/abs/1305.3014)
- 问题定义：广告主设置 targeting 后，需要快速估计 reach/performance trade-off。
- 输入数据：历史广告/用户样本、targeting criteria、分层维度。
- 方法：小规模代表性样本驻留内存，多变量分层采样，针对少数群体做 fuzzy fallback。
- 评估方式：与 uniform sampling、simple stratified sampling 比较速度和估计精度。
- 关键结论：高维受众估计要优先解决查询实时性和长尾覆盖。
- 落地启发：对流量预测的售前/库存查询层，采样与 fallback 是比端到端模型更基础的能力。
- 局限：核心是 reach estimation，不直接解决未来时序变化和 pacing 控制。

### 3.2 Forecasting Granular Audience Size for Online Advertising

- 来源：[arXiv:1901.02412](https://arxiv.org/abs/1901.02412)
- 问题定义：预测细粒度 categorical attribute 组合下的 audience size。
- 输入数据：多个 categorical 属性组合及历史规模。
- 方法：改造 Eclat 频繁项集挖掘，再用时序分析与条件概率近似预测 frequent/infrequent itemsets。
- 评估方式：模拟数据压力测试和真实数据 holdout，对比包括神经网络在内的 baseline。
- 关键结论：频繁项集 + 条件概率比纯神经网络更适合许多高维稀疏组合。
- 落地启发：对定向组合流量预测，先把组合拆解和 fallback 机制做好，再叠加模型。
- 局限：对实时竞价价格、预算控制、策略反馈没有覆盖。

### 3.3 Real-Time Device Reach Forecasting Using HLL and MinHash Data Sketches

- 来源：[arXiv:2502.14785](https://arxiv.org/abs/2502.14785)
- 问题定义：在用户指定多维 targeting 时实时预测设备 reach，避免慢 SQL join。
- 输入数据：多维设备/targeting 数据，集合聚合与交集查询。
- 方法：HyperLogLog 与 MinHash sketch，扩展多层聚合和交集计算，并用 SIMD 优化。
- 评估方式：与传统离线预测系统对比速度和误差。
- 关键结论：集合基数估计是广告 reach 预测的核心基础设施之一。
- 落地启发：如果任务需要频控、去重 reach、跨维度交集，sketch 比明细 join 更适合线上服务。
- 局限：sketch 解决集合估计，不直接学习季节性或投放策略变化。

### 3.4 Ad Serving Using a Compact Allocation Plan

- 来源：[arXiv:1203.3593](https://arxiv.org/abs/1203.3593)
- 问题定义：在 guaranteed contracts 下，每个用户访问到来时选择哪个匹配合约，确保到期交付。
- 输入数据：合约 targeting、需求量、到期时间、未来 user visit forecast。
- 方法：离线生成 compact allocation plan，线上 ad server 低延迟查询，且无状态、易分布式。
- 评估方式：真实 user visits 与 guaranteed contracts。
- 关键结论：预测要转化为可执行的分配计划；线上系统不能依赖复杂全局重算。
- 落地启发：流量预测最好输出给 allocation/pacing 可消费的结构，而不只是日报式预测值。
- 局限：依赖离线 forecast 质量，对突发 spike 和实时反馈讨论有限。

### 3.5 Online Allocation with Traffic Spikes

- 来源：[arXiv:1711.05764](https://arxiv.org/abs/1711.05764)
- 问题定义：线上 allocation 中同时存在可预测随机流量和不可预测 adversarial traffic spikes。
- 输入数据：预测可描述的随机流量，以及未预知 spike。
- 方法：混合 stochastic/adversarial 模型，算法同时利用预测并对预测失准保持鲁棒。
- 评估方式：理论边界与 hardness results。
- 关键结论：纯随机假设会低估真实广告流量的不确定性。
- 落地启发：模型监控应单独识别 spike；遇到 spike 时策略要能降级，而不是强行相信平滑时序。
- 局限：偏理论，不直接给出现成工业特征工程。

### 3.6 Online Ad Allocation with Predictions

- 来源：[arXiv:2302.01827](https://arxiv.org/abs/2302.01827)
- 问题定义：在 display ads/generalized assignment 中利用机器学习预测提升 online allocation。
- 输入数据：未来 impression/assignment 相关预测。
- 方法：learning-augmented online algorithm，要求预测准确时优于 worst-case，预测差时仍稳健。
- 评估方式：合成与真实数据，对比无预测 worst-case 算法。
- 关键结论：预测模型的价值要通过下游 allocation regret/收益衡量。
- 落地启发：上线新流量预测模型时，应做 decision replay，而不仅比较 WAPE/RMSE。
- 局限：不解决预测模型本身如何训练。

### 3.7 Display Advertising with RTB and Behavioural Targeting

- 来源：[arXiv:1610.03013](https://arxiv.org/abs/1610.03013)
- 问题定义：系统性梳理 RTB 机制、用户响应、bid landscape、出价和优化问题。
- 输入数据：用户行为、广告请求、竞价、价格、点击/转化等。
- 方法：综述性质，连接信息检索、机器学习、优化和广告系统。
- 评估方式：非单一实验论文。
- 关键结论：RTB 预测任务天然是多模块联动，不能把流量、价格、响应割裂。
- 落地启发：做剩余流量预测时，应至少知道下游是按 win opportunity、cost opportunity 还是 value opportunity 使用。
- 局限：综述较早，不能代表最新模型效果。

### 3.8 Deep Landscape Forecasting for Real-time Bidding Advertising

- 来源：[arXiv:1905.03028](https://arxiv.org/abs/1905.03028)
- 问题定义：预测每次广告竞价的 market price 分布。
- 输入数据：auction context、bid price、win/loss、可见/不可见 winning price。
- 方法：RNN 建模价格空间上的条件赢标概率，结合 survival analysis 处理 censorship。
- 评估方式：两个大规模真实数据集，多指标对比。
- 关键结论：market price 分布可以不依赖固定参数分布假设，用深度分布模型更灵活。
- 落地启发：如果用预测指导出价，输出 win-rate 曲线比分散的 CPC 点预测更有用。
- 局限：依赖充分的 auction-level 数据，对纯 publisher 侧库存预测不直接适用。

### 3.9 Scalable Bid Landscape Forecasting in Real-time Bidding

- 来源：[arXiv:2001.06587](https://arxiv.org/abs/2001.06587)
- 问题定义：在 second-price auction 中准确学习 winning price 分布。
- 输入数据：赢标时可见 winning price，输标时只有下界的 censored 样本。
- 方法：heteroscedastic fully parametric censored regression 与 mixture density censored network。
- 评估方式：公开 winning price 数据集和大型 DSP 线上/离线评估。
- 关键结论：真实 winning price 往往不满足同方差、单峰、固定分布假设。
- 落地启发：成本/赢量预测要保留异方差和多峰可能性，尤其是行业、媒体、时段混合后。
- 局限：训练和校准复杂度高于普通回归。

### 3.10 A Field Guide for Pacing Budget and ROS Constraints

- 来源：[arXiv:2302.08530](https://arxiv.org/abs/2302.08530)
- 问题定义：预算 pacing 与 ROS pacing 如何协调，以优化广告主回报并满足约束。
- 输入数据：预算、价值、ROS 约束、pacing multiplier、竞价机会。
- 方法：比较 fully-decoupled sequential、min-pacing、fully-coupled dual-based 方法。
- 评估方式：理论分析和基于大型广告平台数据的半合成实验。
- 关键结论：简单串行解耦可能违反约束，min-pacing 在保证和效果上接近 fully-coupled。
- 落地启发：流量预测服务 MPC 时，要把预算和 ROI/ROS 控制状态作为特征与评估维度。
- 局限：重点是 pacing 算法，不是流量预测模型。

### 3.11 Dynamic Budget Throttling in Repeated Second-Price Auctions

- 来源：[arXiv:2207.04690](https://arxiv.org/abs/2207.04690)
- 问题定义：广告主通过 throttling 选择参与部分 auction，以控制总花费。
- 输入数据：预算、auction 序列、估值、second-price 机制。
- 方法：理论分析 throttling 的 regret 和 competitive ratio，提出 OGD-CB。
- 评估方式：随机估值和 adversarial 估值下的理论结果。
- 关键结论：pacing 在随机场景下通常更优；adversarial 场景下 throttling 也可接近最优。
- 落地启发：当流量预测不可靠或竞价环境 adversarial 时，控制策略要具备保守 fallback。
- 局限：理论化较强，对具体广告特征和系统延迟覆盖有限。

### 3.12 Feedback Control for Small Budget Pacing

- 来源：[arXiv:2509.25429](https://arxiv.org/abs/2509.25429)
- 问题定义：小预算 campaign 下 pacing 容易因为 auction 离散和噪声出现不稳定。
- 输入数据：目标 spend rate、实际 spend、预算桶、反馈状态。
- 方法：bucketized hysteresis 与 proportional feedback controller。
- 评估方式：真实 auction 实验，对 pacing accuracy 与 multiplier volatility 做比较。
- 关键结论：控制理论能减少人工调参和小预算震荡。
- 落地启发：对于低流量任务，预测层应输出更强平滑、更宽区间，并让控制层显式处理不确定性。
- 局限：关注 pacing 控制，不处理高维库存预测。

### 3.13 Online Advertising Revenue Forecasting

- 来源：[arXiv:2111.08840](https://arxiv.org/abs/2111.08840)
- 问题定义：预测 publisher 在线广告收入，帮助媒体管理变现策略。
- 输入数据：publisher 自身收入、其他 publisher 收入、静态与时间协变量。
- 方法：Temporal Fusion Transformer，多 horizon 预测和可解释分析。
- 评估方式：多预测 horizon，对比深度时序 benchmark。
- 关键结论：跨 publisher/市场协变量能提升收入预测。
- 落地启发：流量预测应增加大盘、相似任务、行业分组协变量，而不是只看单任务历史。
- 局限：收入受流量、价格和填充率共同影响，不等同于自然请求量。

### 3.14 Interpretable Deep Learning for Forecasting Online Advertising Costs

- 来源：[arXiv:2302.05762](https://arxiv.org/abs/2302.05762)
- 问题定义：预测在线广告市场日均 CPC。
- 输入数据：广告主自身 CPC、竞争对手 CPC pattern、聚类协变量、时间特征。
- 方法：统计模型、机器学习、深度学习和 TFT 对比，并解释特征重要性与 temporal attention。
- 评估方式：长期成本预测，包含 COVID-19 等市场冲击。
- 关键结论：竞争环境协变量能增强成本预测和市场 shift 下的鲁棒性。
- 落地启发：MPC 若依赖未来成本/可赢量，必须把竞价环境变化纳入，而不能只预测请求量。
- 局限：目标是 CPC，不是 impression/request。

### 3.15 Feature Representation Learning for CTR Prediction

- 来源：[arXiv:2302.02241](https://arxiv.org/abs/2302.02241)
- 问题定义：CTR 预测中如何表示高维稀疏特征。
- 输入数据：用户、广告、上下文、交互等 categorical/sparse 特征。
- 方法：综述 feature representation learning，按表示哪些特征、如何表示特征分类。
- 评估方式：综述论文。
- 关键结论：embedding/representation 是广告预测模型的核心。
- 落地启发：流量模型也有大量 sparse id 和组合特征，可借鉴 embedding、hashing、特征交叉，但目标函数要按流量/供给定义。
- 局限：CTR 是响应预测，不能直接替代流量预测。

### 3.16 Temporal Fusion Transformers

- 来源：[arXiv:1912.09363](https://arxiv.org/abs/1912.09363)
- 问题定义：多 horizon 时间序列预测，混合静态特征、已知未来输入和历史外生变量。
- 输入数据：static covariates、known future inputs、observed historical inputs。
- 方法：LSTM 局部处理、自注意力长期依赖、变量选择网络和 gating。
- 评估方式：多个真实数据集 benchmark，并展示解释性用例。
- 关键结论：TFT 适合复杂协变量下的多步预测和解释。
- 落地启发：若要预测 48 个 time slice 或剩余曲线，TFT 是强 baseline。
- 局限：需要足够训练样本和严格 backtesting；不能自动解决控制污染。

### 3.17 PatchTST

- 来源：[arXiv:2211.14730](https://arxiv.org/abs/2211.14730)
- 问题定义：长周期多变量时序预测。
- 输入数据：多通道时间序列。
- 方法：将时间序列切为 patch 作为 token，并采用 channel-independence 共享模型。
- 评估方式：长周期预测 benchmark。
- 关键结论：patching 降低 attention 成本，让模型看更长历史。
- 落地启发：适合有大量相似任务曲线、需要长历史和多 horizon 输出的场景。
- 局限：广告任务的高维 categorical 条件、预算状态和 censoring 仍需额外处理。

### 3.18 Chronos 与 TimesFM

- 来源：[Chronos arXiv:2403.07815](https://arxiv.org/abs/2403.07815)、[TimesFM arXiv:2310.10688](https://arxiv.org/abs/2310.10688)
- 问题定义：用预训练/基础模型支持通用时序预测。
- 输入数据：多领域时序语料，或下游任务历史序列。
- 方法：Chronos 将数值时序 tokenize；TimesFM 使用 decoder-only patched attention。
- 评估方式：跨公开数据集 zero-shot/few-shot benchmark。
- 关键结论：foundation model 可以降低冷启动建模成本，但仍要校准到具体业务损失。
- 落地启发：可作为冷启动 baseline、异常对照或自动 benchmark；生产决策仍要加广告领域特征与约束。
- 局限：不是广告领域专用模型，不能直接理解投放状态、竞价机制和预算约束。

## 5. 对当前 MPC/剩余流量预测方案的映射

结合当前仓库中已有的「历史预测 + 当日校正」「剩余流量 P50/WAPE」语境，论文给出的方向可以落到四层。

### 5.1 保留强 baseline，但把它定义成先验

当前类似：

```text
全日/基础流量预测 -> 当日累计比例校正 -> 剩余流量
```

这与论文里的 forecast-as-prior 思路一致。问题不在于这个结构，而在于不能把校正比例当成天然真值。更稳妥的定义是：

```text
R_base(i, d, k) = 历史、日内形状、任务属性给出的剩余流量先验
R_pred = R_base * posterior_correction
```

其中 `posterior_correction` 最好由模型学习，或至少按有效样本量、任务规模、当前时段、控制污染做 shrinkage。

### 5.2 把“当日累计比例”升级成特征，而不是唯一校正公式

建议特征包括：

- 任务累计比例：`C_obs / C_base`。
- 最近窗口比例：最近 1-3 个 time slice 的观测/基础比值。
- 大盘比例：同媒体、同行业、同广告位、同定向簇的累计偏差。
- 有效样本量：`C_base`、`C_obs`、过去稳定性。
- 控制状态：预算是否接近耗尽、pacer 是否压制、审核/素材/投放状态是否变化。
- 数据质量：数据延迟、回流缺口、异常 spike。

这样模型可以学习「什么时候相信当日比例，什么时候相信历史先验，什么时候相信大盘」。

### 5.3 输出分位数，而不是只输出一个数

若离线指标是 WAPE/MAE，P50 是合理主输出。但 MPC 的损失通常不对称：

- 低估剩余流量：可能导致当前过度抢量、预算提前消耗。
- 高估剩余流量：可能导致当前过保守、后续花不出去。

因此建议至少输出：

```text
remaining_traffic_p50
remaining_traffic_p70
remaining_traffic_p80
uncertainty_band / prediction_interval
```

下游可以按风险选择分位数，而不是把预测层固定死在 P50。

### 5.4 用 decision replay 验证模型，而不是只看 WAPE

论文中的 allocation/pacing 主线都指向同一个结论：预测要用下游决策效果验收。建议离线评估分三层：

| 层级 | 指标 | 目的 |
| --- | --- | --- |
| 预测准确性 | WAPE、MAE、分时段误差、P50/P80 calibration | 判断模型是否真的预测准 |
| 稳定性 | 日内误差曲线、低流量任务误差、spike 表现 | 判断模型是否能被控制系统使用 |
| 决策回放 | 预算消耗节奏、ROI/成本约束、欠投/过投、收益 | 判断预测是否改善 MPC/pacing |

## 6. 推荐落地路线

### 6.1 第一阶段：建立可解释、可回放的 baseline

目标是得到稳定先验，而不是追求复杂模型。

- 用同星期、节假日、行业/媒体/广告位分层建立日级总量和日内 shape baseline。
- 对 `req_cnt`、可竞价请求、展示、消耗分别建口径，避免混用自然供给和受控观测。
- 对低流量任务做强平滑：分层均值、相似任务、最小样本门槛、校正上下界。
- 指标采用 remaining WAPE + 分时段 calibration，重点看 `k=0..47` 每个时点的剩余预测。

### 6.2 第二阶段：学习后验校正

目标是替代人工校正公式。

推荐目标：

```text
y(i,d,k) = log((R_true(i,d,k) + eps) / (R_base(i,d,k) + eps))
```

模型输出：

```text
Q50(y | features), Q70(y | features), Q80(y | features)
```

最终：

```text
R_hat_q = R_base * exp(Qq(y | features))
```

可选模型：

- 中小规模、强结构化特征：LightGBM/CatBoost quantile regression。
- 多 horizon 曲线：TFT、PatchTST 或 seq2seq 分位数模型。
- 冷启动对照：Chronos/TimesFM 作为 benchmark，不直接替代业务模型。

### 6.3 第三阶段：加入库存/受众与竞价环境层

当任务粒度进入复杂 targeting 或 RTB 竞价时，单纯剩余请求预测不够。

- 定向组合规模：引入采样、FIM、HLL/MinHash 或离线 cube。
- 可赢量/成本：引入 bid landscape 分布、win-rate curve、censored training。
- 大盘协变量：媒体、行业、广告位、设备、地域、竞价价格、相似任务聚合。
- 异常识别：spike、数据延迟、策略切换、审核/预算状态变更。

### 6.4 第四阶段：与 MPC/pacing 联合评估

最终验收不应停在模型误差。

- 做历史回放：同一批请求、同一预算和约束，用新旧预测驱动 MPC。
- 衡量预算节奏、成本/ROI/ROS、欠投/过投、收益、稳定性。
- 建立 guardrail：当预测不确定性过高、数据延迟、低样本或 spike 时，自动降级到保守策略。
- 监控预测与控制的闭环污染：pacer 压制后的低流量不能直接反馈成供给下降。

## 7. Baseline、特征、指标、监控建议

### 7.1 Baseline

推荐按复杂度递进：

1. Naive：同星期同 time slice 均值/中位数。
2. Seasonal profile：日级总量预测 × 日内 shape。
3. 分层 baseline：行业/媒体/广告位/定向簇的 shrinkage。
4. 残差校正：学习 `log(true/base)` 的分位数。
5. 多 horizon 曲线模型：直接输出未来 time slice 或剩余量分位数。

### 7.2 特征

| 类别 | 特征 |
| --- | --- |
| 历史流量 | 最近 1/3/7/14/28 天总量、同星期、日内 shape、波动率 |
| 当日观测 | 累计偏差、最近窗口偏差、当前 time slice、有效样本量 |
| 分层大盘 | 媒体、广告位、行业、地域、设备、定向簇聚合趋势 |
| 任务属性 | 预算、出价、目标、投放状态、生命周期、新老任务 |
| 控制状态 | pacer multiplier、预算消耗率、是否压速、是否暂停/审核异常 |
| 竞价环境 | win-rate、winning price 分布、CPC/CPM、竞争强度 |
| 日历事件 | 星期、节假日、促销日、重大活动、工作日调休 |
| 数据质量 | 延迟、缺口、异常 spike、埋点/回流版本 |

### 7.3 指标

| 目标 | 指标 |
| --- | --- |
| 点预测 | WAPE、MAE、sMAPE、按剩余真实流量加权误差 |
| 分位数 | pinball loss、P50/P70/P80 calibration、coverage |
| 日内稳定 | 按 `k` 分桶误差、早期/中期/晚期误差、低流量任务误差 |
| 长尾 | 按任务规模、定向组合稀疏度、冷启动天数分桶 |
| 决策 | 预算节奏误差、欠投/过投率、ROI/ROS 违约、MPC replay 收益 |
| 系统 | 预测延迟、可用率、fallback 命中率、异常告警准确率 |

### 7.4 线上监控

- **数据新鲜度**：每个 time slice 的数据延迟和缺口。
- **预测漂移**：预测分布、真实分布、残差分布按任务层级监控。
- **校准漂移**：P50 是否约一半样本在预测以下，P80 是否覆盖约八成。
- **控制污染**：pacer 压制、预算耗尽、暂停审核样本单独标记。
- **大盘异常**：媒体/广告位/行业分组 spike 与断崖下降。
- **fallback 监控**：长尾、冷启动、低样本任务回退到哪个先验。
- **决策 guardrail**：预测不确定性过高时，MPC 使用保守分位数或降级策略。

## 8. 关键取舍

### 8.1 什么时候用简单 baseline

适合：

- 任务生命周期短、样本少。
- 日内形状稳定，业务先要可解释和可控。
- 下游对错误敏感，需要先建立 guardrail。

不适合：

- 竞价环境快速变化。
- 大量复杂定向组合。
- 有明显跨任务、大盘、行业联动。

### 8.2 什么时候用深度时序/Transformer

适合：

- 有大量相似任务和足够历史。
- 需要多 horizon 输出，如 48 个 time slice 或剩余曲线。
- 静态特征、已知未来特征和历史外生变量都很丰富。
- 业务需要解释变量重要性或 temporal attention。

不适合：

- 长尾组合极多但缺少聚合/采样基础设施。
- 观测强烈受预算/pacer 污染却未标记。
- 只想替代一个简单校正公式，但没有 decision replay。

### 8.3 什么时候用 sketch/FIM/采样

适合：

- targeting 组合查询、reach 去重、交集估计。
- 实时库存查询，明细 join 成本太高。
- 售前预估和频控/去重相关问题。

不适合：

- 需要直接预测竞价价格分布。
- 需要解释未来趋势原因。
- 主要问题是控制策略而非集合估计。

### 8.4 什么时候预测分布而非点估计

必须预测分布的场景：

- RTB winning price 和 win-rate。
- MPC 高估/低估代价不对称。
- 小预算 campaign，离散性和随机波动大。
- 长尾任务，历史样本不足。
- 要做库存承诺或合约风险评估。

## 9. 总结：推荐技术架构

推荐把广告流量预测系统拆成四层：

```text
1. 库存/受众估计层
   sampling / FIM / sketch / historical cube

2. 基础流量先验层
   seasonal baseline / hierarchical smoothing / daily total + intraday shape

3. 后验校正与分位数层
   quantile residual model / TFT / PatchTST / tree-based model

4. 决策消费层
   allocation / pacing / MPC / bid landscape / guardrail
```

最推荐的近期落地路径：

1. 先保留现有「历史预测 + 当日校正」结构，把它定位成 baseline。
2. 建立 `R_base`、`R_true`、`C_obs/C_base`、控制状态、大盘偏差的训练样本。
3. 训练 `log((R_true+eps)/(R_base+eps))` 的 P50/P70/P80 分位数模型。
4. 用 WAPE、分位数校准和 MPC decision replay 三类指标共同验收。
5. 对长尾和 targeting 组合补库存估计层，对 RTB 场景补 bid landscape 分布层。

一句话结论：

> 广告流量预测的核心不是选一个最炫的时序模型，而是把“自然供给、受众/库存、竞价价格、预算控制、当日反馈”拆干净，再用分位数预测和反馈控制把它们接回下游决策。
