# 算法设计文档写作规范

本文档用于指导本仓库后续的算法设计文档写作，尤其是面向“不熟悉当前方案的同事”的说明文档。

## 核心原则

算法设计文档首先要让读者在前 2-3 节内看懂三件事：

1. 当前要解决什么问题。
2. 当前指标和实验结论是什么。
3. 当前方案大概怎么做。

后续章节再展开公式、数据口径、实现细节和推导。不要一开始就堆 SQL 字段、过滤条件、复杂符号或完整数学证明。

## 开头结构

推荐前几节按以下顺序组织：

1. `指标结论`
2. `方案概览`
3. `核心方法` 或具体方法名，例如 `时间片校准`

`指标结论` 只讲最关键的实验结果和指标口径。不要把数据样本、SQL 过滤条件、字段清单放在开头。

`方案概览` 用业务语言解释方案流程，例如：

1. 先做基础预测。
2. 再用最近观测做校准。
3. 最后输出校准后的剩余流量预测。

`核心方法` 只讲当前主方案。如果当前主方案是时间片校准，就重点讲时间片校准，不要把当天累计校准、其他窗口或备选方案并列展开，除非用户明确要求做对比。

## 指标写法

优先用中文解释指标，再给英文缩写。不要默认读者知道 WAPE、MAPE、P50、horizon、tick 等术语。

推荐写法：

> 当前评估指标可以理解为“用当前时间片流量作为权重，计算剩余流量预测的加权相对误差”。

然后再补充：

> 它不是常见的“加权绝对百分比误差”（Weighted Absolute Percentage Error, WAPE），也不是“平均绝对百分比误差”（Mean Absolute Percentage Error, MAPE）。

如果 SQL 字段名和真实口径不完全一致，要明确说明“字段名沿用旧命名，但统计口径已经改变”。不要直接写“WAPE 从 0.90 到 0.70”，应写成“当前 SQL 评估口径下，指标从 0.90 到 0.70”。

## 标题风格

标题要短、直接、名词化。

推荐：

* `指标结论`
* `方案概览`
* `时间片校准`
* `基础预测`
* `平滑公式`
* `评估口径`

避免：

* `先看结论`
* `方案在做什么`
* `为什么要做平滑`
* `为什么用这个公式`
* 其他问句或完整陈述句

## 术语解释

第一次出现关键术语时，用一句话解释。

例如：

* `时间片`：一天被切成固定数量的小时间段，例如 `0-47` 共 48 个时间片。
* `基础预测`：不使用在线观测时，对未来流量的原始预测。
* `校准系数`：用最近观测修正基础预测的倍数。
* `剩余流量`：从当前时间片到当天结束的流量。

术语要统一。比如同一篇文档里不要混用“校正”和“校准”，除非两者有明确不同含义。

## 公式呈现

先给核心公式，再解释符号。公式不要过早展开到完整下标。

推荐先写：

$$
\hat{R}
=
\hat{R}^{base}
\cdot
c
$$

再用表格解释：

| 名词 | 含义 |
| --- | --- |
| $\hat{R}^{base}$ | 基础剩余流量预测。 |
| $c$ | 校准系数。 |
| $\hat{R}$ | 最终剩余流量预测。 |

后续细节章节再展开 $i,d,k,t$ 等下标。

## 数据口径

数据样本、SQL 字段、过滤条件和窗口边界属于细节内容，不要放在文档开头。

只有在它们会影响读者理解指标或方案正确性时，才单独开一节说明。写法应先解释业务含义，再贴字段名。

## 方法取舍

如果用户要求聚焦一个方案，就只详细写这个方案。其他方案只作为实验对照或一句话背景出现。

例如当前流量预测文档中，主方案是最近 `{L}` 个时间片校准，则：

* 开头可以保留当天累计校准的实验值作为对照。
* 方法章节只展开时间片校准。
* 不要再单独写“两种校准窗口”或当天累计校准公式。

## 完成检查

写完算法设计文档后检查：

* 前 2-3 节是否能让新同事理解方案在做什么。
* 开头是否只讲指标和方案概览，没有堆数据细节。
* 标题是否简短直接，不是问句或长句。
* 术语是否先中文解释，再给英文缩写。
* 指标口径是否和 SQL 或实验实际一致。
* 主方案是否聚焦，没有展开无关备选方案。
* 公式是否先简后繁，符号是否紧跟解释。
